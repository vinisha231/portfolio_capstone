Overall Capstone

This page is a reflective narrative on the capstone experience: what it was like to build the Interview Simulator, what I learned about myself, and where I plan to take the project next. Technical details live on the Capstone Experience page; here the focus is on the story of the quarter.

I chose to build an LLM-powered interview simulator because I wanted something that would help people prepare for real technical and behavioral interviews while pushing me to integrate a modern AI API, a structured backend, and a clear user flow. I started by defining the core flow—setup, then question, answer, feedback, then save and optionally follow-up—and only then split the work into frontend, backend, and Bedrock. That decision to lock the flow first kept the project from drifting. The first time the backend returned a generated question and the first time a session showed up on the dashboard became the moments that kept me going when CORS or auth or Bedrock response formats ate hours.

FastAPI with Pydantic and a clear router layout made the API easy to extend; building a fallback when Bedrock failed meant the app stayed usable without AWS credentials, which mattered for demos and for resilience. Rubric-based evaluation gave the frontend something parseable instead of free-form text. I relied heavily on documentation and course resources, reflected in the annotated bibliography, and deploying to Elastic Beanstalk with env-based config made the project feel real and shareable rather than only local.

Bedrock and deployment were where things got difficult. Getting the right request format for Claude versus Llama required reading the docs carefully and handling both in one service; sometimes the model returned JSON with minor formatting issues, so I added fallbacks and validation. That push to make the system degrade gracefully changed how I thought about the rest of the app—I started adding defensive checks and clearer request/response shapes so the frontend and backend stayed in sync. I would have liked more automated tests and a clearer test plan earlier; if I could do it again, I would lock the API contract and write a few integration tests before building out the frontend, and document deployment steps as I went. Time pressure left some ideas (A/B testing prompts, more question banks) as plans rather than shipped, but that kept the scope realistic.

Where I had a chance to work with peers or instructors, feedback on the design spec and status reports helped me tighten the user flow and prioritize. Talking to people who had run or taken interviews clarified what “good” feedback looked like—strength first, then actionable suggestions—and that directly shaped the evaluation rubrics. Explaining the architecture and Bedrock integration to others forced me to simplify my mental model and communicate it clearly, and I got better at estimating how long integration and debugging would take and at asking for help when stuck on AWS or deployment instead of spinning. The habit of “who is this for and what do they need?” stayed central even when there wasn’t formal user testing.

I tended to over-invest in getting one feature right before moving on, which sometimes left less time for polish or tests; in a future project I would time-box spikes and leave buffer for integration and cleanup. The project still left me with a deployable artifact and a narrative I can use to talk about full-stack and AI integration in interviews—and with a clearer sense of how to break a big goal into smaller steps and when to ask for help.

Future Steps

After the course, I plan to keep the Interview Simulator in my portfolio and link to it from my resume and LinkedIn. I will replace the placeholder “Try the app” link with the live URL once the deployed backend and frontend are stable. I may add one or two features (e.g. more interview types or a simple analytics view) if I have time, but the main goal is to maintain it as a working demo I can walk through in conversations. I also want to document the deployment and environment setup so that anyone—or future me—could run or redeploy the app without reverse-engineering the steps. I intend to use this project in job applications and interviews when discussing APIs, cloud services, and LLMs.
