Overall Capstone

This page is a reflective narrative on the capstone experience: what it was like to build the Interview Simulator, what I learned about myself, and where I plan to take the project next. Technical details live on the Capstone Experience page; here the focus is on motivation, challenges, collaboration, and growth.

Why This Project and How I Approached It

I chose to build an LLM-powered interview simulator because I wanted to work on something that would help people prepare for real technical and behavioral interviews while also pushing me to integrate a modern AI API, a structured backend, and a clear user flow. The idea of generating questions and evaluating answers with an LLM felt both useful and technically interesting—it required prompt design, API integration, and a way to persist sessions so users could see their progress.

I approached the project by defining the core flow first: setup (role, company, difficulty, interview type), then question → answer → feedback, then saving the round and optionally asking for a follow-up. From there I split the work into frontend (React), backend (FastAPI), and the Bedrock integration, and used documentation and course resources to fill in gaps as I went. Staying focused on that flow helped me avoid scope creep and ship something usable before adding extras like voice or more interview types.

Motivation came from seeing each piece work: the first time the backend returned a generated question, the first time evaluation produced a score and feedback, and the first time a session showed up on the dashboard. Those milestones kept me going when debugging CORS, auth, or Bedrock response formats felt slow.

What Worked and How It Connects to My Goals

Several things worked well. Using FastAPI with Pydantic made the API clear and easy to extend; the router structure (auth, interview, sessions, dashboard) kept the code organized. Building a fallback when Bedrock failed meant the app stayed usable even without AWS credentials, which was important for demos and for resilience. Rubric-based evaluation (technical, behavioral, design) gave consistent, parseable feedback instead of free-form text, so the frontend could display scores and suggestions reliably.

I relied on outside resources throughout: FastAPI and AWS Bedrock docs, SQLAlchemy and Alembic for the database, and React for the frontend. Those resources are reflected in the annotated bibliography. Connecting the capstone to my goals was straightforward—I want to work on full-stack or backend systems, and this project gave me hands-on experience with REST APIs, cloud services, and LLM integration, which I can point to in a portfolio or in conversations with recruiters.

Deploying the backend to Elastic Beanstalk and wiring environment variables via the console and get-config was a success that made the project feel "real" and shareable, not just local.

Challenges and What I'd Do Differently

The main challenges were around Bedrock and deployment. Getting the right request format for Claude (Messages API) versus Llama (prompt-based) required reading the docs carefully and handling both in the same service. Sometimes the model returned JSON with minor formatting issues, so I added fallbacks and validation. Another challenge was balancing feature work with testing and documentation—I would have liked to add more automated tests and a clearer test plan earlier.

If I could do it again, I would lock the API contract (request/response shapes) earlier and write a few integration tests for the interview flow before building out the frontend. I would also document the deployment steps (EB, env vars, CORS) as I went, so that handoff or future updates would be easier. Missing opportunities included deeper accessibility work on the frontend and more structured user feedback (e.g. a short survey) to validate that the simulator was actually helping people prepare.

Time pressure meant some ideas (e.g. A/B testing prompts, more question banks) stayed as "future plans" rather than shipping in the capstone window, but that helped me keep the scope realistic.

Working with Others and Skills I'll Take Forward

Where I had a chance to work with peers, instructors, or stakeholders, the feedback on the design spec and status reports helped me tighten the user flow and prioritize what to build first. Discussing the simulator with people who had run or taken interviews clarified what "good" feedback looked like (e.g. starting with a strength, then actionable suggestions) and influenced the evaluation rubrics.

I learned that explaining the architecture and the Bedrock integration to others forced me to simplify my mental model and communicate it clearly—a skill I'll use in future roles when documenting systems or onboarding teammates. I also got better at estimating how long integration and debugging would take and at asking for help when stuck on AWS or deployment instead of spinning for too long.

If the capstone had more formal stakeholder involvement (e.g. user testing with job seekers), I would have treated their feedback as primary input for the next iteration; even without that, the habit of "who is this for and what do they need?" stayed central to the design.

Time Management and Growth

Managing time across design, implementation, testing, and documentation was hard. I tended to over-invest in getting one feature "right" (e.g. the evaluation prompt) before moving on, which sometimes left less time for polish or tests. In the future I'll try to time-box spikes and leave buffer for integration and cleanup.

I grew in a few concrete ways: I'm more comfortable reading API docs and adapting examples to my use case; I'm more willing to add fallbacks and defensive checks so the app degrades gracefully; and I care more about clear request/response shapes and error handling so that the frontend and backend stay in sync. I also got better at breaking a big goal (e.g. "support behavioral interviews") into smaller steps (rubric, prompt, JSON parsing, UI) and tackling them one at a time.

With more time, I would add more test coverage, run a small user study, and improve the dashboard (e.g. trends over time, clearer strength/weakness summaries). Even without that, the project left me with a deployable artifact and a narrative I can use to talk about full-stack and AI integration in interviews.

Conclusion / Next Steps

After the course, I plan to keep the Interview Simulator in my portfolio and link to it from my resume and LinkedIn. I'll replace the placeholder "Try the app" link with the live URL once the deployed backend and frontend are stable. I may add one or two features (e.g. more interview types or a simple analytics view) if I have time, but the main goal is to maintain it as a working demo that I can walk through in conversations.

I also want to document the deployment and environment setup so that anyone (or future me) could run or redeploy the app without reverse-engineering the steps. The capstone gave me a concrete project to point to when discussing APIs, cloud services, and LLMs, and I intend to use that in job applications and interviews. Next steps are therefore: link the portfolio to the live app, keep the repo and docs in good shape, and iterate on the simulator when it makes sense alongside other commitments.
